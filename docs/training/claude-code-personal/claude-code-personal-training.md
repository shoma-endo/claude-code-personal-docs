### ③ ターミナルとエディターとは（初心者向け）

本講義では **2 つの道具** を組み合わせて使います。プログラミング未経験でも、役割を押さえれば問題ありません。

#### エディターとは

**エディター**は、PC 上のファイルを開いて読み書きするためのアプリです。メモ帳の高機能版のイメージです。

| 用語 | 意味 |
|---|---|
| フォルダを開く | 講義用の作業場所（プロジェクト）をエディターに読み込む |
| サイドバー（左） | フォルダ内のファイル一覧。Claude が編集したファイル（例: `自己紹介.md`）がここに現れる |
| 編集エリア（中央） | 選んだファイルの中身を表示・編集する |
| 保存 | 変更をディスクに書き込む（多くの場合は自動保存） |

講義で使う VS Code / Cursor / Antigravity IDE は、いずれも **エディター + 内蔵ターミナル** が一体になったツールです。別々のアプリを行き来する必要はありません。

#### ターミナルとは

**ターミナル**は、文字（コマンド）で PC に指示を出す画面です。黒や濃い色の画面に白い文字が並ぶイメージです。

| 用語 | 意味 |
|---|---|
| プロンプト | 入力待ちの行。末尾に `$` や `%`、フォルダ名などが表示される |
| コマンド | 実行したい指示。Enter キーで送信する |
| 出力 | コマンドの結果がその下に表示される |

本講義では、**エディター下部の「ターミナル」パネル** から操作します。

| OS | エディターでターミナルを開く |
|---|---|
| Mac | メニュー「表示」→「ターミナル」、または `` Ctrl + ` `` |
| Windows | メニュー「表示」→「ターミナル」、または `` Ctrl + ` `` |

#### 本講義での役割分担

![エディター、ターミナル、Claude Code が同じフォルダを使い、ターミナルから指示して Claude Code がファイル作成や編集を行う役割分担](/assets/images/training/prep/editor-terminal-claude-roles.png)

| 道具 | あなたがすること | Claude Code との関係 |
|---|---|---|
| **エディター** | フォルダを開く、作られたファイルを開いて内容を確認する | 変更結果を**目で見て確認**する場所 |
| **ターミナル** | `claude` を起動し、日本語で依頼文を入力する | Claude Code と**会話する**場所 |

> [!TIP]
> コマンドは講義資料のとおり **コピー＆ペースト** で始めて構いません。最初から全部入力する必要はありません。貼り付けたら **Enter** を押すのを忘れないでください。

## Session 1 — AIエージェントとは？従来のAIとの違い

**このセッションのゴール** AI エージェントの概念と、従来のチャット型 AI との違いを理解する

### 1. AI は「答える」から「動く」へ

2026 年頃から、AI の役割は大きく変わりつつあります。以前は **質問に答える** ことが中心でしたが、いまは **目標に向かって自律的に動く** ことが当たり前になりつつあります。

| 時期 | イメージ | できること | 人間の役割 |
|---|---|---|---|
| 〜2025（答える AI） | 対話型 AI | 質問に対して回答を返す。テキスト生成が中心 | **実行は人間**が担う |
| 2026〜（動く AI） | AI エージェント | 目標を伝えるだけで、データ収集 → 計画 → 実行 → 検証 → 改善を自律的に進める | **重要箇所の承認**に集中する |

> [!NOTE]
> 「動く」とは、人間がコピペや手作業で全部やるのではなく、AI がファイル操作・コマンド実行・外部ツール連携まで **作業の連鎖** を進めることを指します。本講義の Session 2 以降で触れる Claude Code は、この「動く AI」の代表例です。

![従来型の答えるAIと、計画・実行・検証まで進める動くAIの違い](/assets/images/training/session1/answering-ai-vs-agentic-ai.png)

### 2. 従来型 AI（普通の AI）とは

**ChatGPT・Claude（ブラウザのチャット）** など、いわゆる **対話型 AI** を指します。

#### 特徴

- 1 問 1 答で回答を返す
- テキスト生成が主な役割
- ツールは基本的に使わない（または限定的）
- **実装・実行は人間**が行う

例 「議事録の要約テンプレートを書いて」と依頼 → AI が文章を返す → **コピーしてドキュメントに貼る・整形するのは自分**

#### 具体例で比べる（同じ依頼をしたとき）

「新規事業の案を考えて」と頼んだときの違いです。

| 誰がやるか | 従来型 AI | AI エージェント |
|---|---|---|
| 資料の準備 | **人間**がフォルダからファイルを探し、アップロードする | AI が指定フォルダや URL から集める |
| やり取り | **人間**が追加質問（業界・予算・ターゲットなど）に答える | 目的を伝えたあとは AI が進める |
| 成果物の保存 | **人間**がダウンロードして、決めた場所に保存する | AI がファイル作成・保存まで行う |

従来型 AI は **「答えをもらう」** ところまで。あとは人間の手作業が残ります。

### 3. AI エージェントとは

**目的に合わせて、自分で考えて、自分で行動する AI** です。一度 **目標** を伝えれば、その達成に必要な作業を AI が考えて実行します。本講義で扱う **Claude Code** もこのカテゴリに入ります。

#### 特徴

- 目標を伝えるだけで、計画から実行まで進める
- データ収集 → 計画（Plan）→ 実行（Do）→ 検証（Check）→ 改善（Action）を **自律的に反復** する（いわゆる PDCA を自ら回す）
- ファイル操作・コマンド実行・外部サービス連携など、さまざまな **道具（ツール）** を使い分ける
- 重要な操作だけ **人間が承認** する（要所では人が確認・許可する）

#### 動作イメージ

![AIエージェントがデータ収集、計画、実行、検証、改善を繰り返しながら成果物を作る流れ](/assets/images/training/session1/agent-plan-execute-verify-loop.png)

例 「`議事メモ.txt` を読んで要約を `議事録サマリー.md` に書いて」と依頼 → AI がファイルを読み、要約を書き、保存まで行う → 人間は **結果を確認** する

### 4. 4 つの観点で整理する

「答える」ことと「動く」ことは、根本的に設計が異なります。

| 観点 | 従来型 AI（普通の AI） | AI エージェント |
|---|---|---|
| **入力** | 1 問 1 答の質問 | 目標・ゴール |
| **動作** | 回答を返す | データ収集 → 計画 → 実行 → 検証 → 改善 |
| **ツール** | なし／限定的 | ファイル操作・コマンド実行・外部サービス連携など |
| **人間の役割** | 都度、細かく指示する | 最終承認・境界の設計に集中する |

> [!TIP]
> 覚え方のたとえ 従来型 AI は **賢い検索エンジン**（知りたいことを教えてくれる）、AI エージェントは **優秀な部下**（目的を伝えれば自分で動いて仕上げる）に近い。Session 2 の Claude Code は、この「部下」側。任せ方が変わるので、プロンプトの書き方だけでなく **何を任せ、どこで止めるか** の設計が重要になる（許可ダイアログや安全ルールとつながる）。

### 5. 本講義とのつながり

| この Session で学ぶこと | 次の Session でやること |
|---|---|
| 普通の AI と AI エージェントの違い | Claude Code の起動・ファイル操作・Skills / Hooks |
| 「動く AI」の考え方 | 同梱ファイルの編集・要約・`CLAUDE.md` 追記を実践する |
| 人間は承認・検証に集中 | 許可ダイアログ、`CLAUDE.md`、個人利用の安全ルール |

Session 2 に入る前に、**「文章をもらう AI」ではなく「作業を進める AI」** として Claude Code を捉えておいてください。

## Session 2 — Claude Code の基礎・基本操作と設定

**このセッションのゴール** Claude Code の概要、安全な使い方、ファイル操作、各種コマンド、Skills / Hooks、コンテキスト管理を理解する。

> [!IMPORTANT]
> - **本講義は Claude Code の CLI（ターミナル版）を前提** に進めます。**エディター**でファイルを確認しながら、**ターミナル**（エディター内蔵で可）で指示を出します。
> - `claude` の起動と初回の会話は **事前準備** で済ませてください。

### 1. Claude Code とは

Claude Code は、あなたの PC 上のプロジェクトを相手に、ファイルの読み書きやコマンド実行まで行う **AI エージェント** です。普通のチャット AI が「文章で答える」ことを中心にするのに対し、Claude Code は作業を進めながら結果をファイルやコマンド出力として残します。

#### 本講義の前提 — CLI（ターミナル版）で進める

Anthropic は同じエンジンを、用途に応じて複数の画面で提供しています（[Platforms 公式ドキュメント](https://code.claude.com/docs/en/platforms)）。

| 形態 | 操作のイメージ | 主な用途 |
|---|---|---|
| **CLI（本講義で使用）** | ターミナルで `claude` と入力 | ターミナル作業、定型処理の自動化、プラグイン |
| **Desktop アプリ** | macOS / Windows の専用アプリ | 変更箇所を見比べる確認、複数の作業を並行、画面操作（マウス中心） |
| **VS Code / Cursor 拡張** | エディター内の Claude Code | 変更前後をエディター上で見比べながらの作業 |
| **Web / モバイル** | ブラウザや Claude アプリ | クラウドで継続実行、離席中の依頼・監視など |

本講義では **CLI** を使います。理由は次のとおりです。

- 事前準備でインストールした **`claude` コマンド** が CLI 版である
- **Skills / Hooks / `.claude/`** など、設定をファイルで育てる流れが CLI ワークフローと一致する

> [!NOTE]
> Desktop アプリを普段から使っていても構いません。Desktop と CLI は **同じエンジン** で、同じプロジェクトなら `CLAUDE.md` や `.claude/` の設定を共有できます。違いは主に **操作画面**（ターミナルか GUI か）です。講義中は手順どおりターミナル版で操作し、講義後に Desktop へ移しても学んだ設定は活かせます。

公式の目安 **並列セッションや視覚的なレビュー** には Desktop、**スクリプト・自動化・ターミナル作業** には CLI（[Desktop ドキュメント](https://code.claude.com/docs/en/desktop)）。

### 2. 個人利用の安全ルール

Claude Code はお手元の PC 上で動作しますが、入力した文章と AI の応答は Anthropic のサーバーを経由します（[セキュリティ](https://code.claude.com/docs/ja/security) / [プライバシーポリシー](https://docs.anthropic.com/ja/docs/legal-center/privacy)）。便利さより先に、何を入力してよいかの境界線を決めます。

#### 入力してよい情報

- 講義用のダミーデータ（sample-data フォルダ内のファイル）
- 公開済み情報
- 個人名や会社名を伏せた業務例
- 自分で外部 AI 利用を許可している文書

#### 入力して避けるべき情報

> [!CAUTION]
> 以下の情報は、原則として外部 AI へ送信しないでください。

- 他人の個人情報（本人の同意なく）
- クライアント名、契約金額、未公開案件
- パスワード、API キー、アクセストークン
- 勤務先で外部 AI 利用が禁止されている機密資料

#### ダミーデータ化の例

| 元データ | 置き換え例 |
|---|---|
| 株式会社ABC | クライアントA |
| 田中太郎 | 担当者A |
| 1,250万円 | 約1,000万円 |

### 3. 許可ダイアログ

Claude Code がファイル編集やコマンド実行を行う時、許可を求めることがあります。

![Claude Code の許可ダイアログ。Yes / Yes allow all edits during this session / No の 3 択が表示される](/assets/images/training/session2/permission-dialog.png)

| 選択肢 | 意味 | この段階での目安 |
|---|---|---|
| `Yes` | 今回だけ許可 | 基本はこれ |
| `Yes, allow all edits during this session` | セッション中すべての編集を許可 | 頻出操作だけ慣れてから検討 |
| `No` | 拒否 | 意図しない操作なら選ぶ |

初回は、基本的に `Yes` を選びます。意味が分からない操作が出たら、許可せず質問してください。

### 4. 止め方を覚える

AI を安全に使うために、始め方より先に止め方を覚えます。

| 操作 | 使う場面 |
|---|---|
| `Esc` | 実行中の作業を止めたい |
| `Esc` 2 回 | 直前の編集を取り消したい |
| `/exit` | セッションを終了したい |

### 5. チェックポイント

Claude Code はファイル編集の前に**チェックポイント**を作成します。これは、セッション内で「直前の状態に戻す」ための仕組みです。§4 で覚えた `Esc` 2 回が、この巻き戻し操作の入口になります。

ポイント

- ファイル編集の失敗を戻したい時に使う
- 外部システムへの操作や送信済みの内容は戻せない
- 重要なファイルを扱う時は、作業前にコピーを残しておく

### 6. CLAUDE.md — プロジェクトのメモリ

CLAUDE.md は、Claude Code に対する永続的な指示書です。プロジェクトのルートに置くと、起動時に自動で読み込まれます。

```markdown
# プロジェクトルール

- 常に日本語で応答してください
- 報告書は「ですます調」で作成してください
- 機密情報をファイル名に含めないでください
- 不明点がある場合は、作業前に確認してください
```

ポイント

- 抽象的な精神論より、具体的で検証可能な指示を書く
- 長すぎると毎回の読み込み負荷になるため、200 行以内を目安にする
- 200 行を超えそうなら `.claude/rules/` に用途別で分割する
- 特定フォルダだけに効かせるルールは `paths` で対象を限定できる（触っていないときはコンテキストに載らない）

#### CLAUDE.md のスコープ

| スコープ | 場所 | 範囲 |
|---|---|---|
| このプロジェクト | `./CLAUDE.md` | このプロジェクトのみ |
| 全プロジェクト共通 | `~/.claude/CLAUDE.md` | 自分の全プロジェクト |

> [!NOTE]
> `./CLAUDE.local.md` という個人専用のファイルもありますが、これは「チームと共有したくない個人ルール」を分けるための仕組みです。設定を共有しない個人利用では `./CLAUDE.md` にまとめて問題ありません。

#### CLAUDE.md を書くときのアンチパターン

よく起きる「効かない・逆効果」な書き方と改善例です。

| 書きがちな例 | 問題 | 改善 |
|---|---|---|
| 「丁寧に対応してください」 | 曖昧で検証できない | 「見出しは体言止め、本文はです・ます調で書く」 |
| 「重要な情報は守ってください」 | Claude に何が重要かを委ねている | 送信してはいけない情報を具体的に列挙する |
| 業務マニュアルを丸ごと CLAUDE.md に貼り付ける | 長すぎてコンテキストを圧迫する | `.claude/rules/` に用途別で分割し、必要なときだけ読み込ませる |
| 「状況に応じて判断してください」 | 曖昧な指示は常にデフォルト動作になる | 「判断が必要な場合は、作業前に確認してください」と明示する |

#### サンプルフォルダの `CLAUDE.md` を確認する

エディターで **ルートの `CLAUDE.md`** を開き、中身をざっと読みます。続けて Claude に次を依頼します。

```text
このプロジェクトの CLAUDE.md に書かれているルールを3つ、箇条書きで要約してください。
```

完了の目安

- エディターで `CLAUDE.md` の実物を開けている
- Claude がルールを要約して返す（起動時の自動読み込みを体験）

> [!NOTE]
> **ルールの追記実習**は **§16** で行います（ゼロから作るのではなく、既存の `CLAUDE.md` に 1 行足す）。

### 7. プロジェクトルートと `.claude/` ディレクトリ

Claude Code の設定は、**プロジェクトルート**と **`.claude/` 配下**に置きます。講義用サンプルでは次の構成です。

```text
sample-data/                   # 本講義の作業フォルダ（§5 のあとで準備）
├── 説明書.md                  # サンプルの説明（最初に読む）
├── CLAUDE.md                  # 共有ルール
├── 自己紹介.md                # §14 用（差し替え演習）
├── 議事メモ.txt               # §15 用のダミー議事メモ
└── .claude/
    ├── settings.json          # 権限・フックなど
    └── skills/
        └── meeting-summary/
            └── SKILL.md       # §11 で参照
```

> [!TIP]
> **`.claude` がフォルダ一覧に見えない**ときは、OS とエディターの両方を確認してください。
> - **Mac**：Finder では `Cmd + Shift + .` で隠しファイルの表示を切り替えられます。
> - **Windows**：エクスプローラーの「表示」タブ →「表示」メニュー →「隠しファイル」にチェック（Win10 では「表示」タブ直下の場合もあります）。
> - **VS Code / Cursor**：設定（Mac は `Cmd + ,`、Windows は `Ctrl + ,`）で `files.exclude` を検索し、`.claude` に該当するパターンを削除すると表示されます。

### 8. お願いとガード（CLAUDE.md / settings / hooks）

| 書く場所 | 強制力 | 特徴 | 代表的な使い方 |
|---|---|---|---|
| **CLAUDE.md** | 弱い | Claude が毎セッション最初に読む「行動指針」。努力して守るが絶対ではない | 「報告書は常に日本語で」「命名規則は〜」 |
| **settings.json** | 中〜強 | 権限設定全般。許可・拒否のハードルールを定義する | 危険なコマンドの拒否、許可リスト |
| **hooks** | 非常に強い | 特定イベント発生時に**自動実行**される仕組み。ブロックも可能 | 危険操作の前に自動停止、編集後に自動フォーマット |

![CLAUDE.md、settings.json、hooks の順に強制力が強くなる設定レイヤー](/assets/images/training/session2/config-strength-stack.png)

`CLAUDE.md` は「ソフトなお願い」、`settings.json` は「ハードルール（権限設定）」と覚えると整理しやすいです。同じ内容を両方に書く必要はありません。コンテキストが長くなると `CLAUDE.md` の内容は無視されることがある点に注意してください。

**hooks（フック）** が最も強力です。`PreToolUse` フックを使うとツール実行の直前にチェックを入れ、危険なら自動でブロックできます。

> [!NOTE]
> hooks の設定にはやや技術的な知識が必要です。初心者のうちは「こういう仕組みがある」と知っておくだけで十分で、この講義で実際に設定する必要はありません。

サンプルフォルダの **`.claude/settings.json`** をエディターで開き、中身がほぼ空であることを確認します（講義では触らず「ここに権限設定を書く場所」と理解するだけで十分です）。

> [!NOTE]
> 1 行目の `"$schema": "..."` は、**エディターが入力補完や設定ミスの警告を出すための補助**です。Claude Code の動作には必須ではありません（消しても動きます）。なお `settings.json` は**コメントを書けない形式**なので、メモはこのファイル外（README など）に残します。

#### 設定の優先順位

| 強い ← → 弱い | 内容 |
|---|---|
| 起動時の CLI オプション | 一時的な上書き |
| `.claude/settings.json` | このプロジェクト |
| `~/.claude/settings.json` | 全プロジェクト共通 |

> [!NOTE]
> 許可ダイアログで「常に許可」を選ぶと、`.claude/settings.local.json` という個人用のファイルが自動で作られます。チームで設定を共有しない個人利用では `settings.json` との違いを気にする必要はありません。`.claude/` の中に増えていても問題ないファイルです。

### 9. Plan モード

大きな作業を任せる時は、いきなり編集させず、まず計画だけ作らせます。

| 方法 | 操作 |
|---|---|
| キーボード | プロンプト欄で `Shift + Tab` を 2 回押す |
| 単発指示 | プロンプトの先頭に `/plan` |

Plan モードでは、読み取りと調査だけを行い、編集は承認後に進めます。慣れるまでは、Plan モードを標準にすると安全です。

### 10. プロンプト技法と反復

Claude Code では「一度の指示で完璧な結果を出す」より、**確認しながら段階的に進める**ほうが確実です。

#### 2 段階パターン — 確認してから動かす

Plan モード（§9）と組み合わせ、編集の前に内容を確認します。

```text
まず「何を変更するか」だけ教えてください。まだ編集しないでください。
```

内容を確認したら、続けて

```text
問題なければ進めてください。
```

#### 指示の書き方の原則

| 原則 | 悪い例 | 良い例 |
|---|---|---|
| **具体的に** | 「よくしてください」 | 「見出しを箇条書きに変えてください」 |
| **1 回 1 タスク** | 「全部整理して」 | 「まず `議事メモ.txt` だけを要約してください」 |
| **対象を明示** | 「保存して」 | 「`議事録サマリー.md` として保存してください」 |

#### 期待と違ったときの戻し方

| 状況 | 対応 |
|---|---|
| 一部だけ直したい | 「〇〇の部分だけ、〜に変えてください」と追加指示する |
| やり直したい | `Esc` 2 回でチェックポイントに戻してから再依頼する |
| 最初からやり直したい | `/clear` でリセットし、より具体的な指示で再開する |

> [!TIP]
> うまくいかないときは「指示が足りなかった箇所」を特定して次の依頼に活かすのが上達の近道です。同じ失敗を繰り返すなら、その指示を `CLAUDE.md` に追記することを検討してください。

### 11. Skills / Commands / Hooks

| 部品 | 役割 | 使いどころ |
|---|---|---|
| Skills | 手順・知識のパッケージ（`.claude/skills/各フォルダ/SKILL.md`）。**新規はこちらを推奨** | 議事録整理、資料レビューなど |
| Commands | 短い定型プロンプト（`.claude/commands/*.md`） | `/review` のような 1 行コマンド |
| Hooks | 「〇〇したら自動で△△する」を決めておく | 危険な操作をしようとしたら自動で止める、作業が終わったら音で知らせる |

初回導入では、まず CLAUDE.md から始め、定型手順が必要になったら Skills を追加します。Hooks は「〇〇したら自動で△△する」を決めておく仕組みですが、設定にはやや技術的な知識が必要なので、初心者のうちは「こういう仕組みがある」と知っておくだけで十分です。運用が増えてから段階的に導入します。

#### サンプルの Skill を開いて確認する

エディターで次を開きます。

`.claude/skills/meeting-summary/SKILL.md`

先頭の `description` が「いつこの Skill を使うか」の目印です。続けて Claude に次を依頼します。

```text
meeting-summary という Skill の SKILL.md を読んで、
この Skill が何をするものか、いつ使うべきかを2文で説明してください。
まだ議事メモ.txt は編集しないでください。
```

完了の目安

- Skill ファイルの実物を開けている
- Claude が Skill の目的を説明できる

### 12. MCP（Model Context Protocol）

**MCP（Model Context Protocol）** は、Claude Code と外部ツール・サービスをつなぐための共通のつなぎ方（標準ルール）です。通常の Claude Code は手元のファイルやターミナルを操作できますが、MCP サーバーを追加することで、Slack・カレンダー・Google Drive など外部サービスの情報取得や操作もできるようになります。

#### 仕組みのイメージ

![Claude Code と外部ツールの間に MCP サーバーが入り、通訳・ブリッジとしてチャット、カレンダー、ファイルなどのサービスへつなぐ流れ](/assets/images/training/session2/mcp-bridge-flow.png)

「Slack の #連絡 チャンネルを見て」と指示したとき、MCP サーバーが Slack API を叩いて結果を返します。Claude Code 側は API の仕様を知らなくてよく、MCP サーバーがすべて取り回します。

#### 代表的な MCP サーバーの例

| カテゴリ | サーバー例 | できること |
|---|---|---|
| コミュニケーション | Slack MCP | チャンネル読み取り、メッセージ送信 |
| カレンダー | Google Calendar MCP | 予定の参照・作成 |
| ドキュメント | Notion MCP | ページの検索・作成・更新 |
| クラウドストレージ | Google Drive MCP | ファイルの検索・読み取り |
| Web 検索 | Brave Search MCP | 最新情報の検索・要約 |

Anthropic 公式・コミュニティ製のサーバーがオープンソースで公開されており、自作も可能です。

#### Skills / Hooks との違い

| | 役割 | 変えるもの |
|---|---|---|
| CLAUDE.md / Skills / Hooks | Claude の **ふるまい・手順** をカスタマイズ | Claude 自身の動き方 |
| MCP サーバー | Claude が使える **外部ツール** を追加 | アクセスできる情報・操作範囲 |

「Claude をどう動かすか」が Skills / Hooks の世界、「Claude が何に触れるか」を広げるのが MCP です。

### 13. コンテキストウィンドウ

会話履歴、読み込んだファイル、コマンド結果、CLAUDE.md は、すべて**コンテキストウィンドウ**に載ります。作業机の例えで言えば、机の上に載せられる資料の総量です。載せすぎると重要な情報が埋もれ、返答の質が落ちることがあります。

![会話履歴、ファイル、コマンド出力、CLAUDE.md がコンテキストに載り、増えすぎたら compact や clear で整理するイメージ](/assets/images/training/session2/context-window-desk.png)

#### 対処の流れ

長時間の作業で調子が悪くなったときは、次の流れで対応します。

1. **`/context`** で使用量を確認する
2. 状況に応じてどちらかを選ぶ
   - **同じ作業を続ける** → **`/compact`** で会話を要約圧縮する
   - **別の作業に切り替える** → **`/clear`** で会話をリセットする

#### コンパクション（`/compact`）

会話履歴を要約してコンテキストを空け直す操作です。同じ作業を続けるときに使います。

1. `Esc` でタスクを中断
2. `/compact` を実行
3. 要約が完了したら作業を再開

`/clear` とは異なり、会話の流れを保ったまま容量だけ圧縮できます。

#### 会話をリセットする（`/clear`）

別のタスクに切り替えるときは `/clear` を使います。`CLAUDE.md` やプロジェクトの設定ファイルは残ったままです。

#### セッション再開（`/resume`）

前回のセッションを会話履歴ごと引き継いで再開します。ターミナルを閉じた後でも、前の文脈を復元できます。

#### 使用率の目安

`/context` で表示される使用率を参考に、以下を目安に動きましょう。

| 使用率 | 推奨アクション |
|--------|----------------|
| 〜60%  | 様子見（余裕あり） |
| 60〜80% | `/compact` を検討（予防的に圧縮） |
| 80〜90% | `/compact` を強く推奨 |
| 90% 超 | `/compact` または `/clear` で必ず対処 |

> 60〜70% 付近から「重要な情報の失念」が起き始めるケースもあります。気になったら早めに対処するのが安心です。

#### 日常のコツ（予防）

- 長いファイルを一度に読みすぎない
- 会話が長くなったら `/context` で確認し、`/compact` または `/clear` を検討する
- `CLAUDE.md` が長くなったら `.claude/rules/` 分割や `paths` による限定を検討する

#### モデルによるコンテキスト上限

`/model` で選ぶモデルによって、コンテキストウィンドウ（机の広さ）の上限が異なります。

| モデル | コンテキスト上限（目安） | 日本語文字数の目安 |
|---|---|---|
| Claude Opus 4.7 | 約 1M tokens | 約 50万〜100万文字（Webページ数百〜数千ページ分） |
| Claude Sonnet 4.6 | 約 1M tokens | 約 50万〜100万文字（Webページ数百〜数千ページ分） |
| Claude Haiku 4.5 | 約 200k tokens | 約 10万〜20万文字（Webページ数十〜数百ページ分） |

Haiku は速い反面、上限が小さいため、長い会話や大量のファイル読み込みでは早めに `/compact` が必要になることがあります。

#### モデルの選び方（`/model`）

ターミナルで `/model` を実行すると、利用中のモデルを切り替えられます。

| モデル | 向いている場面 | 速度 |
|---|---|---|
| **Opus 4.7** | 複雑な設計、多ファイル改修、判断が難しい依頼 | 遅め |
| **Sonnet 4.6** | 日常の大半（要約、下書き、Skills 作成など） | 速め |
| **Haiku 4.5** | 単純な整形、短文生成、定型処理 | 最速 |

迷ったら **Sonnet** から始めてください。うまくいかない・複雑なタスクは **Opus**、単純で速さ優先なら **Haiku** に切り替えます。

> [!NOTE]
> 本講義の課金は **Claude Pro プラン以上** のサブスクリプションです。Anthropic 公式のトークン単価表（API 課金）は Claude Code の個人利用とは別の仕組みです。利用量の目安は **付録 B** のレート制限表示、または Claude のアカウント設定を参照してください。
>
> モデルの詳細比較は [Anthropic 公式 — モデル概要](https://docs.anthropic.com/ja/docs/about-claude/models/overview) を参照してください（API 開発者向けの情報が中心です）。

**Part A — AI 秘書（cc-secretary）**（§1〜§7）

### 1. AI 秘書とは

**AI 秘書**（cc-secretary）は、タスク、メモ、予定、振り返りを Claude Code 上で扱うための仕組みです。目的は、単なる TODO 管理ではありません。日々の業務情報をファイルとして残し、後から Claude Code と一緒に整理・改善できる状態にすることです。

Session 2 で学んだファイル操作・CLAUDE.md・Skills の知識を前提に、「自分専用の秘書環境」を立ち上げます。

![素の Claude Code では秘書ワークのフォルダ構成やルールを自分で設計する必要があり、cc-secretary を入れると .secretary にタスク、メモ、アイデア、ミーティング、振り返りが整理される比較](/assets/images/training/session3/cc-secretary-overview.png)

#### 素の Claude Code との違い

素の Claude Code でも「メモを `.md` に書く」「タスクを管理する」ことは可能です。ただし、その場合は **フォルダ構成・命名規則・運用ルールをすべて自分で設計し、CLAUDE.md に書き起こす** 必要があります。cc-secretary は、その「秘書としての型」をプリセットで用意してくれるプラグインです。

| 観点 | 素の Claude Code | cc-secretary を入れた AI 秘書 |
|---|---|---|
| 保存先・フォルダ構成 | 自分で決めて運用する | `.secretary/` 配下を自動生成 |
| カテゴリ（todos / ideas / meetings 等） | 自分で設計する | 14 カテゴリがプリセット |
| 起動・呼び出し | 毎回プロンプトで指示する | `/secretary` コマンドで管理モードへ |
| 運用ルール | 自分で CLAUDE.md に書く | オンボーディングで対話的に生成 |
| 週次レビュー・振り返り | 自分で仕組みを作る | 振り返り導線が組み込み済み |

つまり cc-secretary は、Session 2 で手で作っていた「秘書ワークの土台」を **インストール一発で用意する** ためのプラグインです。手動でも同じことはできるため、ネットワーク制限などで導入できない場合は §7 の手動版 AI 秘書で同等の運用に切り替えられます。

### 2. cc-secretary をインストールする

AI 秘書を使うには、まず cc-secretary プラグインを導入します。Claude Code のプロンプト欄で、1 行ずつ実行します。

```text
/plugin marketplace add Shin-sibainu/cc-secretary
```

続けて

```text
/plugin install secretary@cc-secretary
```

プラグインを有効化するため、再起動します。

```text
/exit
```

ターミナルで再度

```bash
claude
```

完了の目安

- `/secretary` が認識される
- エラーなく AI 秘書（管理モード）が起動する

### 3. AI 秘書でできること

cc-secretary は、Claude Code 上で動く業務秘書プラグインです。対話的なオンボーディングを通じて `.secretary/` フォルダを自動生成し、タスク・メモ・アイデア・週次レビューを一元管理できます。

#### 対応カテゴリ（14 種類）

| カテゴリ名 | 説明 |
|---|---|
| todos | デイリータスク管理 |
| inbox | 思いついたことをすぐ書き留める場所（常に含む） |
| ideas | アイデアの記録 |
| meetings | 議事録 |
| projects | プロジェクト管理 |
| clients | クライアント管理 |
| research | リサーチ・調査 |
| knowledge | ナレッジベース |
| reviews | 週次・月次レビュー（常に含む） |
| content-plan | コンテンツ企画（ブログ/YouTube/SNS） |
| journal | 日記・ジャーナル |
| reading-list | 読書リスト |
| debugging | デバッグログ |
| finances | 財務・経理 |

### 4. オンボーディング

```text
/secretary
```

AI 秘書からヒアリングが始まります。回答例

```text
個人でWeb制作とコンサルをしています。
午前は制作、午後は打ち合わせと資料作成が中心です。
管理したい領域は、タスク、メモ、アイデア、プロジェクト管理です。
```

完了の目安

- `.secretary/` が生成される
- 業務カテゴリや inbox が確認できる

### 5. 基本操作

```text
/secretary
```

タスクを追加

```text
タスク追加 来週の打ち合わせ資料を準備する
```

今日のタスクを確認

```text
今日のタスク
```

メモを取り込む

```text
メモ クライアントA の要望 納期を2週間前倒ししたい。要件の再確認が必要。
```

完了の目安

- タスクが追加される
- 今日のタスクが表示される
- メモが inbox に入る

### 6. 自分仕様に育てる

カテゴリを追加します。

```text
カテゴリ追加 顧客対応
```

`.secretary/` を開き、構造を確認します。

- カテゴリごとのフォルダ
- `inbox/`
- `reviews/`
- タスクやメモの保存先

続けて、業務ルールを追記します。書く場所は **作業フォルダ直下の `CLAUDE.md`**（`.secretary/` と同じ階層）です。Claude Code は起動時にこの `CLAUDE.md` を自動で読み込むので、`/secretary` を呼び出したときにもルールが反映されます。Session 2 の **sample-data** で編集した `CLAUDE.md` をコピーして使っても、新規に作っても構いません。

```markdown
# AI 秘書ルール

- タスクは「高・中・低」の優先度で分類する
- 顧客名は実名ではなく「クライアントA」のように匿名化する
- 週次レビューでは、完了・未完了・来週の重点を分ける
```

### 7. 代替ワーク — プラグインが使えない場合

ネットワークや端末設定により cc-secretary が使えない場合は、手動版 AI 秘書を作ります。これは §1「素の Claude Code との違い」で示した「秘書ワークの土台を自分で設計する」アプローチを、最小構成で再現するものです。

```text
このフォルダに manual-secretary というフォルダを作ってください。
中に以下のファイルを作成してください。

- tasks.md
- inbox.md
- weekly-review.md
- rules.md

tasks.md には「今日」「今週」「いつか」の3区分を作ってください。
inbox.md はメモ置き場にしてください。
weekly-review.md は週次振り返り用のテンプレートにしてください。
rules.md には秘書運用ルールを書いてください。
```

完了の目安

- `manual-secretary/` が作成される
- プラグインなしでも秘書ワークを継続できる

**Part B — Lark CLI**（§8〜§14）

### 8. Lark CLI とは

> [!IMPORTANT]
> - **§1〜§7 で cc-secretary（AI 秘書）のインストールと基本操作が終わっていることを確認してから** 進めてください。Lark CLI は **cc-secretary の後** にインストールします。
> - Lark CLI はあなたの Lark 権限の範囲で API を実行します。**`--as user`** で本人認証し、他人のデータを無断操作しないでください。
> - 機密情報は Session 2 と同様、入力前に匿名化してください。

[Lark CLI](https://github.com/larksuite/cli)（`lark-cli`）は、Lark の **公式 CLI ツール** です。人間がターミナルから直接使うだけでなく、**Claude Code などの AI Agent** が Lark の各種 API を安全に呼び出すために設計されています。

![Claude Code が自然言語の依頼を受け、Lark CLI を実行して Lark クラウド上のカレンダー、タスク、Docs、Base などの業務データを扱う流れ](/assets/images/training/session3/lark-cli-cloud-flow.svg)

| 特徴 | 内容 |
|---|---|
| カバー範囲 | メッセンジャー、Docs、Base、Sheets、カレンダー、メール、タスク、会議など **18 業務ドメイン** |
| コマンド数 | **200+** の厳選コマンド |
| AI 向け Skill | **26** の Agent Skill（`lark-calendar`、`lark-doc`、`lark-base` など） |
| ライセンス | MIT（`npm install` で利用開始可能） |
| 設計思想 | Agent 向けに最適化（簡潔な引数、スマートデフォルト、構造化出力） |

cc-secretary が **ローカルファイル** で秘書業務を担うのに対し、Lark CLI は **Lark クラウド上の業務データ**（予定、タスク、ドキュメント、表など）にアクセスします。講義では **Lark CLI を標準手段** として使います。

### 9. Lark CLI のインストール

#### 前提環境

| 項目 | 要件 |
|---|---|
| Node.js | `npm` / `npx` が使えること |
| Lark アカウント | Lark Suite にログインできること |

> [!NOTE]
> **Node.js が入っていない場合**
>
> ターミナルで `node -v` を実行してバージョンが表示されなければ未インストールです。
> [Node.js 公式サイト](https://nodejs.org) から **LTS 版**をインストールしてください（`npm` も同時に使えるようになります）。
> インストール後、ターミナルを再起動してから `node -v` で確認できます。

#### インストール

```bash
npx @larksuite/cli@latest install
```

#### Agent Skill のインストール（Claude Code 連携用）

Claude Code から Lark を操作するには、公式リポジトリ付属の **Agent Skills** を入れておきます。npm インストール後、次を実行します。

```bash
npx skills add larksuite/cli -y -g
```

> [!NOTE]
> `skills` は Claude Code の **Skill 機構**（Session 2 で触れた、Claude に追加能力を持たせる仕組み）を扱う公式 CLI です。`npx skills ...` で都度実行できるため事前インストールは不要で、ここでは GitHub リポジトリ `larksuite/cli` が同梱する Skill 群（`lark-calendar`、`lark-doc` など 26 種）を取得しています。cc-secretary が `/plugin` で配布されるのに対し、Lark CLI の Skill はこの `skills add` で配布される、と覚えてください。

`-g` はグローバル配置（どのプロジェクトからも参照可）です。講義用プロジェクトだけに置く場合は `-g` を外し、プロジェクト直下で実行してください。

#### インストール確認

```bash
lark-cli --version
```

バージョンが表示されれば OK です。

### 10. Lark CLI の設定

インストール後、**アプリ作成・認証情報の登録** → **ユーザー OAuth ログイン** の 2 段階が必要です。

#### 10-1. アプリ作成・認証情報の登録（初回のみ）

**カスタムアプリを持っていない場合（新規作成）**

```bash
lark-cli config init --new
```

コマンドを実行するとブラウザが開き、アプリ作成〜認証情報の取得まで案内されます。完了すると `app_id` / `app_secret` がローカルに保存されます。

**既存のカスタムアプリがある場合（App ID / App Secret を手元に用意）**

```bash
lark-cli config init
```

対話形式でモードを選択するプロンプトが表示されます。既存アプリの `App ID` と `App Secret` を入力して登録してください。

> 手順の詳細は [公式 Quick Start](https://github.com/larksuite/cli#installation--quick-start) を参照してください。

#### 10-2. ユーザーログイン（OAuth）

自分の Lark データにアクセスするため、ユーザー本人としてログインします。

```bash
lark-cli auth login --recommend
```

`--recommend` は、よく使うスコープをまとめて承認するオプションです。講義ではこれを使うのが簡単です。

その他のログイン例

```bash
# 対話式（ドメイン・権限を TUI で選択）
lark-cli auth login

# ドメインを絞る（カレンダーとタスクだけ、など）
lark-cli auth login --domain calendar,task

# 特定スコープを指定
lark-cli auth login --scope "calendar:calendar:read"
```

#### 10-3. ログイン状態の確認

```bash
lark-cli auth status
```

現在のログインユーザーと付与済みスコープが表示されれば成功です。

#### 認証コマンド一覧

| コマンド | 動作 |
|---|---|
| `auth login` | OAuth ログイン（スコープ付与） |
| `auth logout` | ログアウト・認証情報削除 |
| `auth status` | ログイン状態・スコープ確認 |
| `auth check` | 特定スコープの有無確認（0=OK、1=不足） |
| `auth scopes` | アプリで利用可能なスコープ一覧 |
| `auth list` | 認証済みユーザー一覧 |

#### ユーザー / Bot の使い分け

| 種別 | フラグ | 用途 |
|---|---|---|
| ユーザー | `--as user`（デフォルト想定） | 自分のカレンダー、Docs、タスクなど |
| アプリ（Bot） | `--as bot` | アプリ名義のメッセージ送信など |

講義では **個人の Lark データ** を扱うため、基本は **`--as user`** です。

#### 3 層のコマンド体系

### 13. Claude Code から Lark CLI を使う

講義のゴールは「ターミナルで打つ」だけでなく、**Claude Code が Lark CLI を実行して業務を進める** ことです。

#### 連携の流れ

1. §1〜§7 で **cc-secretary（AI 秘書）を先に完了** する
2. §9 で `lark-cli` と Agent Skills をインストール
3. §10 で `config init --new`（新規アプリ）または `config init`（既存アプリ）と `auth login --recommend` を完了
4. 自然言語で依頼 → Claude が `lark-cli` コマンドまたは Skill を実行

> [!NOTE]
> Claude Code に設定を任せる場合は、バックグラウンドで `lark-cli config init --new` を実行し、表示された **認証 URL をそのまま** ブラウザで開いてもらう運用も可能です（[公式 AI Agent Quick Start](https://github.com/larksuite/cli#quick-start-ai-agent) と同じ手順）。URL は改変せずコピーしてください。OAuth ログイン（`auth login --recommend`）も同様です。

#### Claude Code への依頼例

```text
lark-cli calendar +agenda を実行して、今日の予定を表形式で要約してください。
```

```text
Lark の未完了タスクを確認し、期限が近いものを3件ピックアップしてください。
```

```text
「Claude Code 講義メモ」というタイトルで Lark Docs を新規作成し、
本日の学習要点を3行で書き込んでください。
```

完了の目安

- `lark-cli auth status` が成功する
- `calendar +agenda` で予定が表示される（または Claude 経由で同等の結果が得られる）
- Docs または Base の操作を 1 回以上体験できる

> [!CAUTION]
> Lark CLI は AI Agent から自動実行されるため、プロンプトインジェクション等のリスクがあります。公式 README の [Security](https://github.com/larksuite/cli#security--risk-warnings-read-before-use) を読み、デフォルトのセキュリティ設定を安易に緩めないでください。

### 14. AI 秘書と Lark CLI の使い分け

§1 の比較表が **「素の Claude Code」vs「cc-secretary」** という *導入するかどうか* の判断材料だったのに対し、ここでは **「cc-secretary」vs「Lark CLI」** という *どちらの道具で何をやるか* の使い分けを整理します。

| 用途 | AI 秘書（cc-secretary） | Lark CLI |
|---|---|---|
| タスク・メモの日常的な書き留め | ◎ `.secretary/` に蓄積 | △ Lark タスクと連携する場合 |
| 週次振り返り・ローカル整理 | ◎ | △ |
| カレンダー・会議 | △ | ◎ `calendar +agenda` など |
| チーム共有の Docs / Base | △ | ◎ |
| オフライン・ローカル優先 | ◎ | × API 必須 |

両方を併用する例 朝は `lark-cli calendar +agenda` で今日の予定を確認し、作業中のメモは AI 秘書の inbox に入れ、週末に `週次レビュー` と Lark のタスク一覧を照らし合わせる。

## 付録

### A. `/config` コマンドで設定を対話的に変更する

ターミナルで `/config` と入力すると、Claude Code の設定を対話メニューから変更できます。主な設定項目を以下に示します。

| 設定項目 | 内容 |
|---|---|
| **Auto-compact** | コンテキストが上限に近づいたとき自動で圧縮する |
| **Thinking mode** | Extended Thinking（深い推論）を有効化する |
| **Rewind code (checkpoints)** | ファイル編集履歴を記録し、好きな時点に巻き戻せる |
| **Default permission mode** | ツール実行時の確認レベルを設定する |
| **Language** | 応答言語を指定する（例：Japanese） |
| **Theme** | Dark / Light を切り替える |
| **Teammate mode** | 複数インスタンスの並列処理方法を選択する |

設定は **グローバル → リポジトリ → ローカル** の 3 層で管理され、ローカル設定が最優先です（§10 の設定レイヤーと同じ仕組みです）。

> **参考記事**
>
> [Claude Code の /config で設定できること全31項目を調査してみた（Qiita）](https://qiita.com/suikyo2023/items/dbe53fbff5d2f52070a8)
>
> バイナリ解析により全設定項目を網羅した詳細な調査記事です。v2.1.138 時点の情報です。

### B. ステータスライン表示ツール：cat-claude-statusline

ターミナル下部にステータスラインを表示し、以下の情報をリアルタイムで確認できるツールです

- **コンテキスト使用率**（`/context` を打たなくても常に見える）
- **サブスクリプションのレート制限**（5 時間・1 週間単位の残量）

コンテキストを手動で管理している方に特におすすめです。

> **導入方法**
>
> Claude Code に以下の URL を伝えてインストールを依頼するだけです
>

## トラブルシューティング

### Claude Code 全般

インストール・サインインで困ったときは **事前準備 §①** と [インストールのトラブルシューティング（日本語）](https://code.claude.com/docs/ja/troubleshoot-install) を参照してください。

| 症状 | 原因候補 | 対応 |
|---|---|---|
| 許可ダイアログで迷う | 操作内容が理解できていない | `Deny` して質問する |
| ファイルが更新・作成されない | 依頼文に「保存して」がない | 依頼文を具体化する |
| 内容が期待と違う | 編集前確認が不足 | 「まだ編集せず、改善案だけ出して」と戻る |
| 返答が鈍くなる | コンテキストが満杯 | `/context` → `/compact` または `/clear` |
| 別フォルダで作業している | エディターで開いているフォルダが違う | エディターで **`sample-data` フォルダ** を開き直してから再開 |

### AI 秘書 / Lark CLI

| 症状 | 対応 |
|---|---|
| `/plugin` が使えない | `claude --version` でバージョン確認 |
| プラグインを導入できない | §7 の手動版 AI 秘書に切り替える |
| `/secretary` が認識されない | `/exit` → `claude` で再起動 |
| `.secretary/` が見つからない | 作業フォルダを確認する |
| 実名を入れてしまった | すぐに匿名化して保存し直す |
| `lark-cli: command not found` | `npx @larksuite/cli@latest install` を再実行 |
| Lark で権限エラー | `lark-cli auth login --recommend` を再実行 |
| Lark の操作が失敗する | `lark-cli auth status` と `--as user` を確認 |
| `config init` が進まない | 表示 URL を改変せずブラウザで開く |
